# 升级开发文档

这份文档用于指导 **FlashShield 在已完成 P0 Demo 基础上的下一阶段升级**。

这次升级的目标不是把项目直接推到完整 P1，而是做一个 **P0 进阶版 / P0+**：

- 保留当前已经跑通的跨链自动保护主链路
- 把 B 链保护动作从“固定 80% mock 资产切换”升级为“更接近真实对冲语义的 mock short”
- 把保护规模从固定值升级为基于公式的动态值

这次升级 **不建议** 直接引入 GMX、Hyperliquid、真实 DEX swap 或自动买回。

------

## 1. 升级目标

### 核心目标

把当前项目从：

- “A 链风险事件 -> Reactive -> B 链固定 80% 保护”

升级为：

- “A 链风险事件 -> Reactive -> B 链按公式自动开一笔 mock short 对冲头寸”

也就是说，这次升级最重要的是让项目叙事更贴近最初设想：

- A 链仓位接近清算
- Reactive 捕捉到爆仓前兆
- B 链自动建立反向对冲仓位
- B 链盈利逻辑用于覆盖 A 链损失

------

## 2. 为什么这次升级不直接做真实协议接入

### 不直接接 GMX 的原因

GMX 更适合未来产品化方向，但当前阶段直接接入会引入：

- 外部协议接口理解成本
- 测试网联调不确定性
- 资金路径与保证金模型复杂度
- 写路径失败处理

这会明显提高 bug 风险，并可能破坏当前已经稳定的 P0 Demo。

### 不直接接 Hyperliquid 的原因

Hyperliquid 更不适合当前这版纯 EVM callback 架构直接接入。后续如果真要对接，通常还要引入 agent / relayer 层。

### 不直接做预言机 / Health Factor 重构的原因

这类升级是合理的，但它会影响 A 链触发源逻辑，不如先把已经跑通的 B 链执行语义升级好。当前阶段应优先升级 **B 链金融表达能力**，而不是同时动两端主路径。

------

## 3. 本次升级范围

### 本次升级要做

#### 1) 把固定 80% 改成动态对冲公式

当前 B 链执行逻辑是：

- 触发后把 `100 / 0` 切成 `20 / 80`

升级后应变成：

- 根据一组输入参数计算 `hedgeSize`

推荐公式原型：

```text
Hedge_Size =
  Collateral_Value_A * (Price_trigger - Price_target)
  / Contract_Multiplier_B
```

这里的目标不是追求金融建模绝对正确，而是让 Demo 具备：

- 明确的对冲计算逻辑
- 非固定比例
- 可解释的参数来源

#### 2) 把 B 链动作语义改成 mock short

当前 B 链动作更像：

- 风险资产切换到稳定资产

升级后应更像：

- 自动开一笔 mock short

也就是说，B 链状态应当更接近：

- `positionDirection = SHORT`
- `hedgeSize`
- `entryPrice`
- `triggerPrice`
- `targetPrice`
- `positionStatus`

而不是继续以 `riskTokenBalance / stableTokenBalance` 作为主要展示语义。

#### 3) 前端展示同步升级

前端应从“保护比例 / 稳定资产比例”转为展示：

- 当前策略是否已触发对冲
- mock short 是否已建立
- 对冲规模是多少
- 触发价 / 目标价是多少
- 该规模为何这样计算

------

## 4. 本次升级明确不做

### 不做真实 GMX 接入

当前只保留“未来可替换为 GMX”的接口边界，不直接调用 GMX。

### 不做真实 Hyperliquid 接入

当前架构不适合直接原样接 Hyperliquid。

### 不做自动买回

自动买回仍然属于 P1。

### 不做真实 DEX swap

本次升级重点是“mock short 语义”而不是“现货卖出再买回”。

### 不重写 A 链触发源

仍然允许先使用现有 A 链风险模拟器与手动价格推进，只要 B 链升级完成即可。

------

## 5. 推荐的升级后组件语义

### A 链：PositionRiskSimulator

继续扮演：

- 风险仓位模拟器
- 风险事件产生器

可选增加但不强依赖：

- `collateralValue`
- `targetPrice`

### Reactive：ReactiveProtection

继续扮演：

- 风险事件监听器
- 跨链 callback 驱动器

它的主职责不变，尽量少改主路径。

### B 链：ProtectionExecutor -> 更接近 Hedging Executor

建议保留文件名不一定改，但语义升级为：

- 根据传入参数计算 hedge size
- 建立一笔 mock short 状态
- 发出 `ShortPositionOpened` 或等价事件

------

## 6. 推荐的数据结构变化

### 当前状态的问题

当前状态更偏资产切换语义：

- `riskTokenBalance`
- `stableTokenBalance`
- `amountProtected`

### 升级后推荐状态

建议升级为：

- `strategyId`
- `positionDirection`
- `hedgeSize`
- `triggerPrice`
- `targetPrice`
- `collateralValue`
- `contractMultiplier`
- `positionStatus`

如需兼容旧展示，可保留旧字段，但前端主展示口径应切换到 short 语义。

------

## 7. 技术风险评估

### 中低风险

- 公式引入
- 状态字段重构
- 前端展示改造

### 中风险

- 参数单位统一
- BigInt 计算与整数截断
- 旧状态与新状态兼容

### 当前不应碰的高风险项

- GMX 真接入
- Hyperliquid 真接入
- A 链预言机重构
- 自动买回

------

## 8. 分支建议

这次升级 **建议新开分支**。

原因：

- 当前 `main` 已经是可提交 demo
- 这次升级属于增强版，不应污染稳定演示线

推荐分支名：

- `codex/p0-plus-hedge`

------

## 9. Sub-agents 建议

这次升级默认 **不需要** 以 sub-agents 为主模式。

原因：

- 合约公式
- B 链状态结构
- 前端展示字段

这三者高度耦合，最好由一个主线程统一收口。

如果一定要拆，最多只建议：

- 主 agent：B 链合约与总体集成
- sub-agent：前端展示字段和文案适配

但默认仍建议单线程推进。

------

## 10. 一句话结论

这次升级最正确的方向不是“做一个更真的交易所”，而是：

**把当前已经跑通的自动保护 Demo，升级成更接近“自动反向对冲”语义的 Demo。**
