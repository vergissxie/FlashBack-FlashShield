# FlashShield 开发步骤

这份文档不是替代[开发文档](开发文档.md)，而是把开发文档里的方案转成**可直接执行的开发顺序**。  
结论先写在最前面：**开发时不要只盯着开发文档直接写代码，应同时维护一份开发步骤文档。**  
原因很简单：开发文档负责定义“做什么”，开发步骤负责定义“先做什么、后做什么、如何验收”。

------

## 0. 当前 P0 目标

在黑客松周期内，完成一条真实可验证的跨链自动保护链路：

1. A 链风控模拟合约产生风险事件
2. Reactive Contract 监听该事件
3. Reactive 自动 callback 到 B 链
4. B 链执行合约完成一次动态 hedge 计算，并建立一笔 mock short
5. 前端展示“风险触发 -> 自动保护 -> 结果对比”

P0 明确不做：

- 真实交易所
- 真实永续协议集成
- 自动恢复
- 真实协议成交
- 完整产品化后台

------

## 1. 开发前冻结项

开始写代码前，先把下面这些常量冻结，不要边写边改：

### 1.1 链路定义

- A 链（origin）：Ethereum Sepolia
- Reactive 运行环境：Reactive Lasna
- B 链（destination）：Base Sepolia

### 1.2 触发规则

- 只让 `NearLiquidation` 触发保护动作
- `LiquidationTriggered` 仅用于展示或对照，不进入 P0 主路径

### 1.3 B 链保护动作

- 当前版本采用 mock short 对冲
- 不接真实外部协议
- 不做自动恢复
- 默认示例逻辑：
  - 读取 `collateralValue`
  - 使用当前 `triggerPrice`
  - 结合 `targetPrice`
  - 按 `contractMultiplier` 自动计算 `hedgeSize`
  - 结果写成一笔 `direction = SHORT` 的 mock 对冲头寸

### 1.5 P1 预留位

- `RecoveryTriggered` 属于 P1，不进入 P0 主路径
- 自动恢复 / 自动平对冲头寸属于 P1，不进入 P0 或 P0 加强版
- P0 编码时要为 P1 留扩展位，但不实现完整恢复逻辑

### 1.4 前端展示口径

前端只展示四件事：

- A 链价格变化
- A 链风险状态变化
- Reactive callback 已触发
- B 链保护结果与“未保护 vs 已保护”对比

------

## 2. 开发顺序

开发必须按下面顺序推进，不要并行乱写。

### 阶段 1：A 链风控模拟合约

目标：先做出一个稳定的风险事件源。

要完成的内容：

- 编写 `PositionRiskSimulator`
- 定义仓位结构体
- 定义 `entryPrice`、`markPrice`、`liquidationThreshold`、`status`
- 编写 `updateMarkPrice()`
- 在条件满足时发出：
  - `NearLiquidation`
  - `LiquidationTriggered`

验收标准：

- 本地测试能验证价格推进后事件正常触发
- 至少覆盖“未触发 / 接近清算 / 已触发”三种状态

### 阶段 2：B 链执行合约

目标：先把目标链保护执行器写稳。

要完成的内容：

- 编写 `ProtectionExecutor`
- 定义策略状态：
  - `hedgeSize`
  - `triggerPrice`
  - `targetPrice`
  - `direction`
  - `status`
- 编写 `executeProtection()`
- 加入 callback 来源校验
- 发出 `ShortPositionOpened`
- 预留未来 `executeRecovery()` 的扩展空间，但 P0 不实现

验收标准：

- 本地测试能验证执行前后状态变化
- 重复执行时有明确限制
- 来源校验失败时能正确 revert

### 阶段 3：Reactive Contract

目标：把跨链自动触发打通。

要完成的内容：

- 基于官方 demo 骨架实现 Reactive Contract
- 订阅 A 链 `NearLiquidation`
- 在 `react()` 中解析事件
- callback 到 B 链 `ProtectionExecutor`

验收标准：

- 能正确订阅 A 链事件
- callback 参数格式与 B 链合约对齐
- 能通过来源校验完成一次回调执行

### 阶段 4：部署与真链路联调

目标：在真实测试网上打通最小闭环。

要完成的内容：

- 部署 A 链合约
- 部署 B 链合约
- 部署 Reactive Contract
- 完成订阅配置
- 手动调用 A 链 `updateMarkPrice()`
- 验证 Reactive 是否 callback 到 B 链

验收标准：

- 区块浏览器可看到 A 链事件
- 区块浏览器可看到 B 链执行结果
- 至少跑通一次完整链路

### 阶段 5：前端最小可用页

目标：把真实链路包装成可演示的界面。

要完成的内容：

- 钱包连接
- 展示当前策略参数
- “开始模拟”按钮
- 展示 A 链价格推进
- 展示风险状态变化
- 展示 Reactive callback 状态
- 展示 B 链保护结果
- 展示“未保护 vs 已保护”对比

验收标准：

- 不依赖本地假数据完成整套展示
- 页面能从链上读取真实状态和事件

### 阶段 6：演示稳定性加固

目标：让 demo 可重复演示。

要完成的内容：

- 固定一套价格推进脚本
- 固定一套策略初始参数
- 固定一套展示文案
- 补充失败提示
- 准备区块浏览器链接

验收标准：

- 至少完整演练 3 次
- 每次都能在合理时间内跑通

## 5. 当前仍需最终确认的点

这些点不多，但最好现在就拍板：

1. A 链是否固定为 Ethereum Sepolia，B 链是否固定为 Base Sepolia。
2. `NearLiquidation` 是否作为 P0 唯一保护触发事件。
3. B 链 mock short 的 `contractMultiplier` 与输入参数口径是否固定。
4. 前端“未保护 vs 已保护”是否采用同一价格路径下的收益对比展示。

如果这四项都确认，P0 需求就足够清晰，可以直接进入开发。

------

## 6. 开发纪律

为了避免黑客松中后期失控，开发时遵守这些规则：

- 不新增 P0 外功能
- 不临时改链路定义
- 不把前端假数据当链上结果
- 不在主链路未打通前做 UI 美化
- 不在 Reactive callback 未稳定前接真实外部协议
- 不在 P0 主链路未稳定前加入自动恢复

------

## 7. Sub-Agents 使用建议

这个项目适合使用 sub-agents，但要用于**模块化并行开发**，不要用于真链路联调阶段的无序并行。

推荐分工：

### Sub-agent A：A 链合约

负责：

- `PositionRiskSimulator`
- 风险状态与事件定义
- A 链单元测试

### Sub-agent B：B 链合约

负责：

- `ProtectionExecutor`
- `hedgeSize / triggerPrice / targetPrice / direction` 状态写入
- callback 来源校验
- B 链单元测试

### Sub-agent C：前端

负责：

- 钱包连接
- 状态展示
- “开始模拟”按钮
- “未保护 vs 已保护”对比

### 主 Agent

负责：

- Reactive Contract
- callback 参数对齐
- 部署与联调
- 最终集成与问题收口

使用规则：

- 合约接口先冻结，再让 sub-agents 并行实现
- 真链路联调阶段尽量由主 Agent 主控
- 不要让多个 sub-agents 同时修改同一份核心合约
- 每个 sub-agent 负责自己的文件边界

------

## 8. 一句话执行策略

**先打通真链路，再做前端包装；先保证自动化成立，再考虑真实性增强。**
