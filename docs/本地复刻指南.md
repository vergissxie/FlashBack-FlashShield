# FlashShield 本地复刻指南

这份文档面向第一次拿到仓库的人，目标是让你即使不熟悉区块链，也能按步骤把 FlashShield 拉到本地、装好依赖、部署合约、启动前端，并完成一次最小演示。

这份指南默认你使用：

- Ubuntu / WSL2
- Node.js 20.x
- npm 10.x
- 浏览器钱包，例如 MetaMask

---

## 1. 拉取仓库

SSH 方式：

```bash
git clone git@github.com:SeeMoon357/FlashShield.git
cd FlashShield
```

HTTPS 方式：

```bash
git clone https://github.com/SeeMoon357/FlashShield.git
cd FlashShield
```

### 前端侧依赖

## 3. 准备环境变量

先复制模板：

```bash
cp .env.example .env
```

然后编辑根目录 `.env`，至少填写下面这些值：

```env
ETHEREUM_SEPOLIA_RPC_URL=
BASE_SEPOLIA_RPC_URL=
REACTIVE_RPC_URL=https://lasna-rpc.rnk.dev/
REACTIVE_CHAIN_ID=5318007
REACTIVE_CALLBACK_PROXY=
DEPLOYER_PRIVATE_KEY=
ORIGIN_CHAIN_ID=11155111
DESTINATION_CHAIN_ID=84532
EXPECTED_RVM_ID=
```

还没部署之前，下面这三项可以先留空：

```env
POSITION_RISK_SIMULATOR_ADDRESS=
PROTECTION_EXECUTOR_ADDRESS=
REACTIVE_PROTECTION_ADDRESS=
```

### 每个变量是什么意思

- `ETHEREUM_SEPOLIA_RPC_URL`：A 链 RPC
- `BASE_SEPOLIA_RPC_URL`：B 链 RPC
- `REACTIVE_RPC_URL`：Reactive Lasna RPC
- `REACTIVE_CALLBACK_PROXY`：Reactive 官方 callback proxy 地址
- `DEPLOYER_PRIVATE_KEY`：用于部署和测试网交互的钱包私钥
- `EXPECTED_RVM_ID`：部署 Reactive 合约的钱包地址，不是合约地址

### 如何得到 `EXPECTED_RVM_ID`

如果你准备用 `.env` 里的 `DEPLOYER_PRIVATE_KEY` 来部署 Reactive 合约，那么它对应的钱包地址就是 `EXPECTED_RVM_ID`。

你可以用这条命令算出来：

```bash
node derive-rvm-id.js
```

输出的 `0x...` 地址，填回 `.env`：

```env
EXPECTED_RVM_ID=0x...
```

## 4. 先跑一次本地测试

这一步不是部署，但建议先做，确保工程本身没坏。

```bash
cd contracts
npx hardhat compile
npx hardhat test
cd ..
```

如果测试都通过，说明合约工程本地没问题。

---

## 5. 部署 A 链合约

先部署 `PositionRiskSimulator` 到 Ethereum Sepolia：

```bash
cd contracts
npx hardhat run --network ethereumSepolia scripts/deploy-position-risk-simulator.js
```

成功后你会看到：

- 合约地址
- 部署交易哈希
- 自动生成的部署记录文件

把输出里的地址记下来，并写回根目录 `.env`：

```env
POSITION_RISK_SIMULATOR_ADDRESS=0x...
```

---

## 6. 部署 B 链合约

部署 `ProtectionExecutor` 到 Base Sepolia：

```bash
cd contracts
npx hardhat run --network baseSepolia scripts/deploy-protection-executor.js
```

成功后同样会输出：

- 合约地址
- 部署交易哈希

把地址写回根目录 `.env`：

```env
PROTECTION_EXECUTOR_ADDRESS=0x...
```

### 这个合约的作用

它负责在 B 链接收 Reactive callback，并根据：

- `collateralValue`
- `triggerPrice`
- `targetPrice`
- `contractMultiplier`

自动计算一笔 `hedgeSize`，然后把这次动作写成一笔链上的 `mock short` 对冲头寸。

注意：这是当前 Demo 里的链上 mock short 执行，不是真实 GMX / Hyperliquid / 外部协议成交。

## 8. 给 Reactive 和 callback proxy 补充执行资金

FlashShield 这套 Demo 部署完不代表马上能跑，还需要补充资金和 reserve。

### 8.1 给 Reactive 合约补充 debt 覆盖

```bash
cd contracts
npx hardhat run --network reactiveLasna scripts/cover-reactive-debt.js
```

### 8.2 给 Base callback proxy 充值 reserve

先在终端里设置目标地址：

```bash
export PROXY_ADDRESS="$REACTIVE_CALLBACK_PROXY"
export TARGET_ADDRESS="$PROTECTION_EXECUTOR_ADDRESS"
export DEPOSIT_VALUE_ETH="0.01"
```

然后执行：

```bash
npx hardhat run --network baseSepolia scripts/deposit-to-proxy.js
```

### 8.3 如有需要，直接给合约补原生币

如果你需要给某个合约直接补充余额：

```bash
export TARGET_ADDRESS="$PROTECTION_EXECUTOR_ADDRESS"
export FUND_VALUE_ETH="0.01"
npx hardhat run --network baseSepolia scripts/fund-contract.js
```

## 9. 检查部署是否成功

可以直接运行：

```bash
cd contracts
npx hardhat run --network ethereumSepolia scripts/verify-deployment.js
```

这会检查：

- A 链合约地址是否有代码
- B 链合约地址是否有代码

- 对应部署交易是否存在
- receipt 状态是否成功

## 10. 把部署地址同步到前端

前端读的是 `web/.env.local`，所以部署完后还要再建这个文件。

在 `web/` 目录下新建 `web/.env.local`：

```env
NEXT_PUBLIC_ORIGIN_CHAIN_ID=11155111
NEXT_PUBLIC_DESTINATION_CHAIN_ID=84532
NEXT_PUBLIC_POSITION_RISK_SIMULATOR_ADDRESS=0x...
NEXT_PUBLIC_PROTECTION_EXECUTOR_ADDRESS=0x...
NEXT_PUBLIC_REACTIVE_PROTECTION_ADDRESS=0x...
```

这三项地址要与你刚刚部署出来的地址一致。

## 12. 页面正确使用顺序

页面上的流程分为三段：

1. 开启演示仓位  
   这一步会在 A 链发起一次真实交易，需要第一次签名。

2. 触发接近清算  
   这一步会在 A 链发起第二次真实交易，需要第二次签名。

3. Reactive 自动回调到 B 链  
   这一步不需要第三次签名，系统会自动完成跨链保护。

## 13. 推荐的演示参数

第一次建议用这组：

- 策略 ID：`FS-DEMO-01`
- 开仓价格：`100`
- 清算阈值：`88`
- 目标触发价：`90`

注意：

- `清算阈值` 必须小于 `开仓价格`
- `目标触发价` 应该低于 `开仓价格`

## 15. 如何判断整条链路跑通了

你应看到这些现象：

### A 链侧

- 仓位成功创建
- 当前价格推进到接近清算区间
- 页面或脚本能确认 `NearLiquidation` 已发生

### Reactive 侧

- 不需要第三次签名
- Reactive 自动监听并发起 callback

### B 链侧

- 保护结果自动刷新
- 当前策略成功时，会写入一笔 `mock short` 状态

也就是：

- 对冲方向：`SHORT`
- 对冲规模：`hedgeSize`
- 状态：`ShortOpened`

---

## 16. 如果页面没有自动更新

依次检查：

1. 当前策略 ID 是否就是你刚刚开的那个
2. 是否等待了 10 到 20 秒
3. 钱包是否接在正确网络
4. 根目录 `.env` 与 `web/.env.local` 地址是否一致
5. `REACTIVE_CALLBACK_PROXY` 和 `EXPECTED_RVM_ID` 是否填对

## 17. 当前版本的边界

本仓库当前是可提交的黑客松 Demo，而不是完整产品。当前边界包括：

- B 链保护动作是链上 mock short 执行，不是真实 GMX / Hyperliquid / 外部协议成交
- 当前对冲规模由公式计算，但仍然不是外部真实永续协议头寸
- 还没有自动恢复 / 平掉 mock short
- 还没有完整存入 / 取回资金池逻辑

这些边界不会影响“跨链自动保护链路已经成立”这个核心结论。
