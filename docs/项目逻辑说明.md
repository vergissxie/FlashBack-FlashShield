# FlashShield 项目逻辑说明

这份文档用于帮助队友、评委或后来接手项目的人快速理解 FlashShield 的产品逻辑。

它解释的是：

- 这个项目现在到底能干什么
- 它在产品层面是什么逻辑
- 用户和评委应该怎么理解这条链路
- 当前 Demo 做到了哪里，没做到哪里

它**不是**代码实现文档，但适合作为“这个项目到底在做什么”的业务说明材料。

---

## 1. 一句话理解项目

FlashShield 是一个“跨链风险自动保护” Demo。

它的核心意思是：

- 当 A 链上的某个仓位进入危险区
- Reactive 会自动监听这个风险事件
- 然后在 B 链自动执行一个保护动作

所以我们要表达的不是“我们做了一个交易所”，而是：

**我们证明了 Reactive 可以把一条链上的风险信号，自动转化成另一条链上的保护行为。**

---

## 2. 当前 Demo 里的三条链路角色

### A 链：Origin 链

当前是 Ethereum Sepolia。

它的职责是：

- 模拟一个仓位
- 接收价格变化
- 当价格进入危险区时，发出风险事件

所以 A 链是“风险源头”。

它负责告诉系统：

- 这个仓位已经危险了
- 应该启动保护动作了

### Reactive：中间自动化层

当前是 Reactive Lasna。

它的职责是：

- 监听 A 链发出来的 `NearLiquidation`
- 把这个事件转成 callback
- 把 callback 发到 B 链

所以 Reactive 是“自动化转发器”。

它不是 UI，也不是策略页面，而是整个系统自动化成立的关键。

### B 链：Destination 链

当前是 Base Sepolia。

它的职责是：

- 接收 Reactive callback
- 执行保护动作
- 把执行结果写进链上状态

所以 B 链是“保护动作执行地”。

---

## 3. 当前 Demo 的完整人脑逻辑

当前逻辑可以这样理解：

1. 用户在前端发起一个 A 链演示仓位
2. 这个仓位在 A 链上有开仓价格、清算阈值、抵押价值和 B 链对冲目标价
3. 用户再在前端触发一次“接近清算”
4. A 链发出 `NearLiquidation`
5. Reactive 监听到这个事件
6. Reactive 自动向 B 链发送 callback
7. B 链执行合约收到 callback
8. B 链根据 `collateralValue / triggerPrice / targetPrice / contractMultiplier` 自动计算 `hedgeSize`
9. B 链把这次动作记成一笔 `mock short`
10. 前端读取 B 链状态，展示“已对冲 / short 已建立”

所以真正要强调的是：

- 用户只负责在 A 链发起风险相关动作
- 从风险事件发生开始，后半段是自动的

也就是：

- `A 链风险事件`：用户触发
- `Reactive 监听与转发`：自动
- `B 链保护动作`：自动

---

## 4. 当前 Demo 里所谓的“保护动作”到底是什么

这是最容易让人误解的地方。

当前 Demo 里，B 链的保护动作不是：

- 真正去 GMX / Hyperliquid 上开永续空单
- 也不是真实现货减仓或真实协议撮合

当前做的是：

- 在 B 链执行合约里维护一份 mock short 状态
- 当保护动作触发后，合约会把以下结果写进链上：
  - `hedgeSize`
  - `triggerPrice`
  - `targetPrice`
  - `direction = SHORT`

所以现在的保护动作是：

**链上可验证的 mock short 对冲执行**

不是：

**真实协议成交**

但重点是：

- 这个 `hedgeSize / triggerPrice / direction` 不是前端本地变量
- 它们是 B 链合约真实写出来的链上状态

所以当前 Demo 的表达应该是：

**我们已经实现了“真实跨链自动保护链路”，但当前保护动作还是 mock short 执行，不是真实永续协议成交。**

---

## 5. 当前 Demo 已经做到的事情

### 已经做到

- A 链仓位创建是真实链上交易
- A 链风险事件触发是真实链上交易
- Reactive 监听和 callback 是真实跨链自动化
- B 链保护结果是真实链上状态
- 前端能连接钱包并驱动整条主链路
- 前端能展示当前策略是否真的已经在 B 链完成保护

### 当前已经成立的核心价值

就是这句话：

**源链风险一出现，目标链保护动作会自动发生。**

这条价值已经不是概念，而是已经被真实测试网验证。

---

## 6. 当前 Demo 还没做的事情

### 1. 还没有真实永续协议下单

当前 B 链没有真的去：

- GMX 开一笔真实 short
- Hyperliquid 开一笔真实 short
- 或其他真实链上衍生品协议建立仓位

原因不是做不到，而是黑客松阶段先优先验证 Reactive 主链路和跨链自动化成立。

### 2. 还没有真实预言机 / 健康度模型

当前 A 链的风险判断仍然是演示型的：

- 用模拟仓位表达风险
- 用前端 / 脚本推进价格
- 在阈值附近触发 `NearLiquidation`

它还不是：

- 真实借贷协议里的 health factor
- 真实 oracle + 借贷仓位联动

### 3. 还没有真实资金池或保证金模型

当前没有完整实现：

- 用户在 B 链存入真实保证金
- 用真实保证金去开真实空单
- 头寸结束后再取回剩余资产

现在更多是“对冲头寸状态模拟”。

### 4. 当前对冲规模由公式自动计算

也就是说：

- 当前策略结果不再固定为某个百分比
- 程序会根据 `collateralValue / triggerPrice / targetPrice / contractMultiplier` 自动算出 `hedgeSize`

这不是前端假写，而是当前链上策略规则自动计算的结果。

---

## 7. 当前 Demo 最准确的定位

我们现在不应该把它讲成：

- 完整跨链交易系统
- 完整跨链对冲协议
- 完整自动买卖平台

更准确的讲法应该是：

**一个基于 Reactive 的跨链风险自动响应 Demo。**

或者更直白一点：

**它展示的是“跨链自动保护能力”，不是“完整交易产品”。**

---

## 8. 对外说明时的推荐口径

可以直接这样说：

> FlashShield 现在的 Demo 是这样的：A 链负责产生风险事件，Reactive 负责自动监听并跨链转发，B 链负责自动建立一笔 mock short 对冲头寸。当前 short 还是 mock 的链上执行，不是真实 GMX / Hyperliquid 下单，但整条跨链自动化链路是真的。

如果别人继续问“那你们现在的 B 链到底做了什么”，可以回答：

> 当前 B 链会根据抵押价值、触发价、目标价和合约乘数，计算一笔应建立的对冲规模，并把它以 mock short 形式写到链上。这一步是链上真实状态变化，不是前端本地变量变化。

如果别人问“为什么还没接真实 GMX / Hyperliquid”，可以回答：

> 因为当前 P0+ 先验证的是 Reactive 的跨链自动保护链路，而不是某个具体永续协议的集成。协议接入属于下一阶段增强项。

---

## 9. 当前最重要的认知边界

无论是队内沟通还是对外答辩，都应统一这几点：

### 我们现在证明的是“自动保护”

不是证明：

- 真正成交深度
- 最优交易路径
- 完整资管策略

### 我们现在是 Demo，不是完整产品

所以：

- mock 执行不等于造假
- 动态 hedge 不等于真实永续协议成交
- 当前重点是证明跨链自动保护链路成立

---

## 10. 一句话总结

FlashShield 当前版本最准确的定义是：

**一个基于 Reactive Network 的跨链风险自动保护 Demo。**

它已经真实证明了：

- A 链风险事件可以被真实触发
- Reactive 可以真实监听并跨链回调
- B 链可以真实写入保护结果

但它暂时还没有扩展到：

- 真实永续协议下单
- 真实 oracle / health factor 触发
- 完整资金池与保证金生命周期
- 自动恢复或更完整策略生命周期

只要主链路是真实、稳定、能讲清楚，就成立。

### 我们现在最大的价值是 Reactive 的自动化能力

整个项目真正的亮点是：

- 不是“一个按钮”
- 不是“页面动画”
- 不是“某个固定百分比”

而是：

**A 链风险事件 -> Reactive 自动监听 -> B 链自动执行保护**

这才是项目最该被看见的东西。

---

## 10. 当前一句话总结

FlashShield 当前已经是一个可提交的黑客松 Demo：

- 主链路真实
- 跨链自动化真实
- B 链结果真实

但它仍然是一个“风险自动保护 Demo”，而不是“完整交易产品”。
