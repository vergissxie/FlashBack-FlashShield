# 开发文档

先约定两个术语，避免后面再混：

- **仓位 / 合约单**：指金融意义上的多单、空单、持仓。
- **智能合约**：指链上的程序代码。

这版方案里，你们**不是在复刻一个真实交易所**，而是在做一个**跨链风险响应层**：A 链上的仓位风险被一个可控的链上风控模拟器表达成事件，Reactive 监听这个事件后，自动在 B 链执行预先授权好的保护动作。Reactive 官方文档确认了它的核心范式：Reactive Contracts 监听 EVM 链上的事件日志，在订阅事件发生时自动执行 Solidity 逻辑，并向目标链发送 callback；它可以在没有用户或机器人再次签名的情况下，基于链上事件自主运行。([Reactive Network](https://dev.reactive.network/reactive-contracts))

另外，这个设计也符合 Reactive 的系统边界：Reactive Contract 同时存在于 RNK 和 ReactVM 两个环境中，但二者**不共享状态**；ReactVM 负责事件处理，RNK 负责用户交互和订阅管理；ReactVM 也不能直接访问外部 RPC 或离线服务。所以，真正的业务状态应尽量落在目标链的执行合约里，而不是依赖 Reactive 内部存储。([Reactive Network](https://dev.reactive.network/reactive-contracts))

在**一天时间、允许 AI 辅助的黑客松**约束下，P0 还需要再做一个关键收敛：

- **不做真实协议路由集成，不做真实成交依赖。**
- **B 链保护动作采用“可验证的 mock short 对冲”模型。**
- 也就是说，B 链执行合约在 callback 到来时，不去真实协议下单，而是根据 `collateralValue / triggerPrice / targetPrice / contractMultiplier` 计算一笔 `hedgeSize`，并把这次对冲动作以 `mock short` 状态和事件写进链上。
- 这样做的目的是把 MVP 的重点放在“跨链事件触发 -> 自动执行保护动作 -> 链上结果可验证”这条主链路，而不是把时间消耗在测试网流动性、协议路由、滑点和外部协议集成问题上。
- 如果黑客松后续时间允许，再把这个 mock 执行器升级为真实协议集成。

## 1. 价值主张与需求分析 (Value Proposition & Requirement Analysis)

### 核心用户目标 (Core User Goal)

为跨链持仓用户提供一个**风险事件触发后的自动保护机制**：当 A 链仓位接近爆仓或进入极端风险区间时，系统自动在 B 链建立一笔反向 mock short 对冲头寸，减少组合继续恶化的损失。

### 功能性需求 (Functional Requirements)

#### P0：黑客松必须完成

- **P0: 用户可以连接钱包并创建一个保护策略。**
  至少配置：A 链仓位方向、开仓价、风险阈值、B 链保护动作、B 链策略地址。
- **P0: A 链存在一个“仓位风险模拟器”智能合约。**
  该合约不是交易所本体，而是用来模拟一个金融仓位的状态，至少包含：`entryPrice`、`markPrice`、`liquidationThreshold`、`status`。
- **P0: A 链风险模拟器可以在价格推进时发出风险事件。**
  推荐事件至少有：`NearLiquidation`、`LiquidationTriggered`。
- **P0: Reactive Contract 订阅 A 链风险事件，并在事件出现时自动 callback 到 B 链。**
  这里不做轮询，不做后端机器人；采用 Reactive 原生的事件驱动模式。([Reactive Network](https://dev.reactive.network/reactive-contracts))
- **P0: B 链存在一个策略执行合约（`ProtectionExecutor`）。**
  它接收 callback 后，自动执行一次保护动作。
- **P0: B 链保护动作采用“mock short 对冲”而不是真实永续开仓，也不依赖真实协议成交。**
  例如：把执行合约内部写入一笔 `direction = SHORT` 的 mock 对冲头寸，并记录 `hedgeSize / triggerPrice / targetPrice / contractMultiplier` 等字段，用来表达“B 链已经建立反向对冲仓位”的自动保护动作。
- **P0: 用户完成首次策略创建后，不需要在触发时再次点击确认。**
  自动执行发生在 B 链策略合约内，而不是临时调起用户钱包二次签名。Reactive 的自动化能力允许基于链上事件直接驱动 callback，但前提是目标链合约和策略参数已提前准备好。([Reactive Network](https://dev.reactive.network/education/module-1/reactive-contracts))
- **P0: B 链执行合约必须校验 callback 来源。**
  至少校验 Callback Proxy 和嵌入的 RVM ID，避免伪造回调。([Reactive Network](https://dev.reactive.network/origins-and-destinations))
- **P0: 前端提供一个“开始模拟”按钮。**
  点击后按预设价格路径分步推进 A 链价格，前端实时展示：A 链风险升级、Reactive 被触发、B 链执行保护动作、以及“未保护 vs 已保护”的结果对比。

#### P1：后续可改进

- **P1: 把 B 链动作从 mock short 升级到真实协议集成。**
  例如在目标链上调用真实永续协议，完成一笔真实 short 的建立。
- **P1: 增加第二条规则，实现自动恢复 / 自动平对冲头寸。**
  例如新增 `RecoveryTriggered` 事件，在风险解除或价格回升后执行恢复。该能力明确属于 P1，不进入 P0 或 P0 加强版。
- **P1: 支持更完整的策略状态机。**
  如：`IDLE -> PROTECTED -> RECOVERED -> CLOSED`。
- **P1: 增加多策略、多仓位、多用户支持。**
- **P1: 增加轻量索引层。**
  用于历史记录、策略列表、收益曲线和事件时间线展示。
- **P1: 增加通知。**
  如 Telegram、邮件、站内提醒。
- **P1: 把“脚本推进价格”升级为“链上价格源/链上场景驱动器”。**

#### P2：长期演进

- **P2: 接入更可信的价格源。**
  例如将 A 链风险判定从人工脚本推进，升级到预言机或更接近真实市场结构的风险输入。
- **P2: 引入策略模板与参数治理。**
  允许不同风险偏好的用户选择不同触发阈值、动作比例和恢复规则。
- **P2: 引入索引、审计与监控支持。**
  为策略历史、收益归因、链路追踪和安全审计提供稳定的数据面。

### 非功能性需求 (Non-Functional Requirements)

- **性能 (Performance):**
  演示过程中，用户点击“开始模拟”后，前端应在 1 到 3 个确认周期内看到 A 链状态推进；风险事件到 B 链动作的链路应控制在 demo 可接受的几十秒量级内，并在前端明确显示“等待 Reactive callback”状态。
- **可扩展性 (Scalability):**
  P0 只支持单策略、单用户、单资产路径；架构上保留未来扩展到多策略和多资产的空间。
- **安全性 (Security):**
  必须做 callback 来源校验、权限隔离、暂停机制、最小授权、非重入保护；若 P0 采用 mock 执行，则仍需保留与未来真实资产流一致的权限边界。
- **可用性 (Availability):**
  即使前端临时出问题，链上策略和执行状态仍可通过区块浏览器与事件日志验证。
- **可维护性 (Maintainability):**
  A 链风险模拟、Reactive 自动化、B 链策略执行必须清晰分层，避免把过多业务状态塞进 Reactive 内部。

## 2. 系统架构设计 (System Architecture)

### 架构选型与论证 (Architecture Selection & Rationale)

推荐采用 **“前端 + 三合约 + 无独立后端”的事件驱动模块化单体架构**。

原因很直接：你们当前要验证的不是一个完整交易所，而是一条清晰的自动化链路——**A 链风险事件 -> Reactive -> B 链保护动作**。在这个阶段引入后端、数据库、任务队列，收益很小，复杂度却会明显上升。Reactive 的官方架构本身就是事件驱动：RC 监听 origin 事件，ReactVM 执行逻辑，destination 链承接 callback；再加上 RNK 与 ReactVM 状态分离、ReactVM 不能直接访问外部 RPC，这更适合把最终业务真相放在 B 链执行合约里。权衡上，这种架构牺牲了一些中后台体验，也牺牲了 P0 的真实成交能力，但能最大化一天内的交付确定性。([Reactive Network](https://dev.reactive.network/reactive-contracts))

### 核心组件与职责 (Core Components & Responsibilities)

#### 1) 前端 DApp

职责：

- 钱包连接
- 策略创建
- “开始模拟”按钮
- 展示 A 链风险状态、Reactive 触发状态、B 链保护结果

#### 2) A 链风控模拟合约（`PositionRiskSimulator`）

职责：

- 存储一个或多个金融仓位对象
- 维护 `entryPrice`、`markPrice`、`liquidationThreshold`、`status`
- 接收价格更新
- 在满足条件时发出 `NearLiquidation` / `LiquidationTriggered`

#### 3) 部署在 Reactive Lasna 上的 Reactive Contract

职责：

- 订阅 A 链风险事件
- 在 `react()` 中判断是否需要动作
- 发 callback 到 B 链执行合约
  Reactive 的标准角色就是监听 origin 事件并驱动 destination callback。([Reactive Network](https://dev.reactive.network/reactive-contracts))

#### 4) B 链执行合约（`ProtectionExecutor`）

职责：

- 接收 callback
- 校验 Callback Proxy 和 RVM ID
- 根据公式计算 `hedgeSize`
- 建立一笔链上可验证的 `mock short`
- 更新策略状态与结果
  目标链 callback 的可验证性和安全性由 Callback Proxy + RVM ID 机制保证。([Reactive Network](https://dev.reactive.network/origins-and-destinations))

#### 5) 场景推进器（`ScenarioRunner`）

职责：

- 不是独立后端
- 可以是前端里的一个“开始模拟”流程，也可以是 Foundry 脚本
- 负责按固定价格路径推进 A 链价格
  例如：`100 -> 92 -> 84 -> 72`

### 推荐链路

- **A 链（origin）**：Ethereum Sepolia
- **Reactive**：Reactive Lasna
- **B 链（destination）**：Base Sepolia

这样选择的原因是：这些测试网都在 Reactive 官方文档列出的 origin/destination 支持范围内；其中 Base Sepolia、Ethereum Sepolia、Lasna 都可作为 destination，适合做标准的跨链 callback 演示。([Reactive Network](https://dev.reactive.network/origins-and-destinations))

这里的角色定义需要固定：

- **A 链 = origin 链**：风险事件产生链。
- **B 链 = destination 链**：Reactive callback 的目标链，也是保护动作执行链。
- **Reactive Lasna**：Reactive Contract 的运行环境，不等同于 A 链或 B 链业务链本身。

### B 链保护动作的推荐实现形式

在当前 P0+ 版本中，推荐把 B 链动作写成一个**链上可验证的 mock short 状态写入**，而不是一开始就写成真实协议成交：

1. `ProtectionExecutor` 内部维护策略状态，例如：
   - `hedgeSize`
   - `triggerPrice`
   - `targetPrice`
   - `direction`
   - `status`
2. 当 Reactive callback 到达时，合约校验来源。
3. 校验通过后，根据 `collateralValue / triggerPrice / targetPrice / contractMultiplier` 自动计算 `hedgeSize`。
4. 合约把这次动作写成 `ShortOpened` 状态，并发出 `ShortPositionOpened(...)` 事件。
5. 前端读取事件和链上状态，展示“已对冲 / mock short 已建立”结果。

这样做的好处是：

- 保护动作是真正发生在 B 链合约上的。
- 结果可以被浏览器和前端直接验证。
- 它比固定比例切仓更贴近“跨链反向对冲”这个原始叙事。
- 未来替换成真实 GMX 或其他永续协议时，前端与 Reactive 主链路几乎不用重做，只替换 `executeProtection()` 的内部实现。

### 事件流（端到端）

1. 用户在前端创建策略
2. 前端点击“开始模拟”
3. 前端按价格脚本调用 A 链 `updateMarkPrice()`
4. A 链仓位进入风险区间，发出 `NearLiquidation`
5. Reactive 监听事件并自动 callback 到 B 链
6. B 链执行一次动态对冲计算，并写入 mock short 状态
7. 前端展示“已保护”状态和结果差异

## 3. 技术栈推荐 (Technology Stack Recommendation)

### 技术选型列表

- **前端 (Frontend):** Next.js + TypeScript + wagmi + viem + Tailwind CSS
- **后端 (Backend):**
  - **P0：无独立后端**
  - **P1：Node.js / NestJS 轻量索引服务（可选）**
- **数据库 (Database):**
  - **P0：无数据库**
  - **P1：PostgreSQL / Supabase（可选）**
- **智能合约开发:** Solidity + Foundry
- **云服务/部署 (Cloud/Deployment):** Vercel（前端）+ GitHub Actions（CI）

### 选型理由 (Rationale for Selection)

- **Next.js + TypeScript**
  适合快速做一个前端主导的 hackathon 项目；页面、交互、钱包连接和演示控制台能放在同一套工程里。
- **wagmi + viem**
  EVM 前端交互非常成熟，适合处理钱包连接、合约调用、事件读取。
- **Foundry**
  最适合这类“合约 + 脚本 + 部署 + 本地测试”一体化工作流。
- **P0 不上后端 / 不上数据库**
  这是一个故意的工程取舍。因为你们现在的核心价值在链上自动化闭环，而不是在数据平台。P0 阶段把复杂度压到最低，能显著提高成功率。
- **P0 不接真实外部协议**
  一天时间里，真实协议集成会把不确定性集中在测试网流动性、接口可用性、授权细节和失败处理上，不利于交付；mock short 对冲更能突出 Reactive 的核心价值。自动恢复同理，也应放在 P1，而不是提前塞进 P0。
- **P1 才考虑索引服务和数据库**
  只有当你们需要历史曲线、事件回放、通知和多策略管理时，数据库与后端才开始真正有价值。

### 关于官方 demo 的使用建议

建议**继续参考 Reactive 官方的 Uniswap stop-loss / take-profit demo，但只借工程骨架，不借业务语义**。官方 demo 的价值在于它已经验证了“Reactive Contract 监听 origin 事件 + callback contract 执行目标动作”的组合方式；尤其是其 callback contract 做最终链上校验、状态管理和执行逻辑，这些都值得借鉴。([GitHub](https://github.com/Reactive-Network/reactive-smart-contract-demos/tree/main/src/demos/uniswap-v2-stop-take-profit-order))

## 4. 开发路线图 (Development Roadmap)

### 第一阶段：MVP（P0）

**目标：**
在黑客松周期内，跑通一条完整的自动保护链路。

**范围：**

- A 链单仓位模拟
- 手动/脚本推进价格
- `NearLiquidation` 事件触发
- Reactive 自动 callback
- B 链执行一次 mock 保护动作
- 前端展示结果

**MVP 的最小成功定义：**
用户点击“开始模拟”后，可以完整看到：

1. A 链价格推进
2. A 链仓位进入 near liquidation
3. Reactive 自动触发
4. B 链策略合约自动执行一次可验证的保护动作
5. 前端显示“未保护 vs 已保护”的差异

### 第二阶段：产品化完善（P1）

**目标：**
让产品从“自动保护 demo”升级为“有完整策略生命周期的原型”。

**范围：**

- 从 mock short 升级到真实永续协议集成
- 增加 `RecoveryTriggered`
- 自动恢复 / 自动平对冲头寸
- 多用户多策略
- 通知与策略历史
- 更像真实产品的链上价格源
- 可选的轻量索引服务与数据库

## 5. 潜在挑战与风险评估 (Challenges & Risks Assessment)

### 技术风险 (Technical Risks)

- **风险 1：把“仓位”和“智能合约”继续混淆。**
  这会导致队内沟通反复跑偏。
- **风险 2：把项目做成“伪交易所”。**
  一旦试图实现完整撮合、真实 perp、真实清算，你们会迅速失控。
- **风险 3：callback 安全校验没做好。**
  这是最实际的链上风险。
- **风险 4：把自动恢复也硬塞进 P0。**
  生命周期过长会显著增加不确定性。
- **风险 5：前端动态演示做得像“假动画”，而不是“真链上变化”。**
  评委会质疑真实度。
- **风险 6：P0 就接真实永续协议，导致协议接口、保证金和测试网环境把主链路拖垮。**
  这会把失败点从 Reactive 主链路转移到外部协议。

### 产品与市场风险 (Product & Market Risks)

- **风险 1：评委误以为你们在做一个完整交易所。**
- **风险 2：评委误以为你们在做“无损防爆仓”。**
- **风险 3：如果 B 链动作定义不清，用户会搞不懂这到底是对冲、减仓还是套利。**

### 缓解策略 (Mitigation Strategies)

- **针对风险 1：**
  对外统一口径：
  **我们不是交易所；我们是跨链风险响应层。**
- **针对风险 2：**
  不讲“救回爆仓”，只讲：
  **在风险事件后自动执行保护动作，减少继续恶化的损失。**
- **针对风险 3：**
  把 B 链动作明确成：
  **在 P0+ 中执行可验证的 mock short 对冲，不把它描述成真实现货卖出或真实永续成交。**
- **针对风险 4：**
  P0 只做“自动保护”；自动恢复放入 P1。
- **针对风险 5：**
  前端展示必须绑定真实链上事件与真实交易哈希，避免纯前端假动画。
- **针对风险 6：**
  P0 先把 B 链保护动作收敛为链上可验证的 mock short；只有在主链路完全稳定后，再尝试替换为真实 GMX / 真实永续协议调用。

## 6. 下一步行动建议 (Actionable Next Steps)

1. **统一项目一句话定义**
   建议固定为：
   **A cross-chain liquidation response demo: when a position on chain A approaches liquidation, Reactive automatically executes a protective action on chain B.**
2. **冻结 P0 业务边界**
   明确：
   - 不做真实交易所
   - 不做真实永续集成
   - 不做自动恢复
   - 不做真实永续协议接入
   - 只做自动保护动作
3. **统一术语表**
   队内以后只说：
   - A 链仓位 / origin 链
   - A 链风控合约
   - Reactive Contract
   - B 链执行合约 / destination 链
   不再混用“链”和“合约”一词。
4. **先搭最小合约闭环**
   开发顺序建议：
   - `PositionRiskSimulator`
