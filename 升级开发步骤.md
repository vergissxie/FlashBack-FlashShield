# 升级开发步骤

这份文档不是替代 [升级开发文档.md](升级开发文档.md)，而是把升级方案转换成 **可以直接执行的开发顺序**。

结论先写在最前面：

**这次升级按“P0+ 动态对冲版”推进，不按“直接接真实协议”推进。**

------

## 0. 本次升级的执行策略

### 只做两件核心事

1. 把固定 80% 保护改成动态 hedge 公式
2. 把 B 链动作语义改成 mock short

### 暂时不做的事

- GMX 真接入
- Hyperliquid 真接入
- 自动买回
- 真实 DEX swap
- 预言机重构

------

## 1. 开始前先做的事

### 1.1 新开分支

推荐分支名：

- `codex/p0-plus-hedge`

原因：

- `main` 当前已是可提交 demo
- 增强版不应直接在稳定线裸改

### 1.2 冻结本次升级边界

开始编码前，先统一这几点：

- A 链继续沿用当前 `PositionRiskSimulator`
- Reactive 主路径尽量少改
- 核心变化集中在 B 链与前端
- 当前升级只做 mock short，不接真实衍生品协议

------

## 2. 开发顺序

### 阶段 1：设计动态对冲输入参数

目标：先把公式参数定义清楚。

最小参数集建议为：

- `collateralValue`
- `triggerPrice`
- `targetPrice`
- `contractMultiplier`

建议：

- `triggerPrice` 可直接沿用当前风险触发价
- `targetPrice` 可作为策略参数新增
- `collateralValue` 可以先用模拟值
- `contractMultiplier` 先固定为策略参数

输出物：

- 参数定义清单
- 单位约定清单

验收标准：

- 所有参数都能明确说出单位和用途
- 前后端对参数命名一致

### 阶段 2：改 B 链状态结构

目标：让 B 链状态从“资产切换”语义转向“short 仓位”语义。

建议新增或替换为：

- `hedgeSize`
- `positionDirection`
- `collateralValue`
- `triggerPrice`
- `targetPrice`
- `contractMultiplier`
- `positionStatus`

建议事件改为：

- `ShortPositionOpened`

输出物：

- 更新后的 `ProtectionExecutor.sol`
- 新事件定义

验收标准：

- B 链链上状态能表达“已开 mock short”
- 前端不必再依赖 `20 / 80` 才能展示保护结果

### 阶段 3：在 B 链实现动态公式

目标：在 callback 到达时按公式计算 `hedgeSize`。

建议：

- 用纯链上整数计算
- 所有输入都在 callback 前准备好
- 对异常值做最小校验

输出物：

- 动态 hedge 公式实现
- 对应单测

验收标准：

- 不同输入参数下，`hedgeSize` 不同
- 不再出现“永远 80%”的固定结果
- 单测覆盖正常值和边界值

### 阶段 4：调整 Reactive callback 参数

目标：让 Reactive 能把 B 链需要的最小参数带过去。

建议：

- 尽量少改 Reactive 主骨架
- 只补足 B 链计算所需最少字段

输出物：

- 更新后的 `ReactiveProtection.sol`

验收标准：

- callback 参数成功对齐
- 不影响当前已跑通的主链路

### 阶段 5：前端展示升级

目标：让前端展示“对冲头寸”，而不是“固定切仓”。

前端应改成展示：

- 当前策略是否已建立对冲
- `hedgeSize`
- `triggerPrice`
- `targetPrice`
- `collateralValue`
- 当前动作：`SHORT_OPENED`

可以保留：

- A 链风险时间线
- Reactive 回调状态

输出物：

- 更新后的 `live-demo-dashboard.tsx`
- 更新后的状态读取逻辑

验收标准：

- 不再主要展示“稳定资产 80%”
- 页面主叙事切换为“自动建立反向对冲头寸”

### 阶段 6：端到端联调

目标：重新跑通一次升级后的真链路。

要完成：

- 部署新 B 链执行器
- 部署新 Reactive 合约（如需要）
- 前端指向新地址
- 跑通一次从 A 到 B 的完整链路

验收标准：

- A 链风险事件正常触发
- Reactive 正常回调
- B 链成功写入 mock short 状态
- 前端成功展示新的 short 语义结果

------

## 3. 风险控制

### 最容易出 bug 的地方

1. 公式单位不一致
2. BigInt 整数除法截断
3. 旧前端仍按旧字段读取
4. callback 参数顺序对不齐

### 控制策略

1. 先写参数和单位表，再写代码
2. 先补 B 链单测，再跑真链路
3. 前端改动放在 B 链状态稳定后进行
4. Reactive 主骨架少改

------

## 4. 时间预估

如果只做这次文档定义的两项核心升级，通常属于：

- **中等工作量**
- 明显低于接 GMX
- 明显低于做真实资金池

粗略看：

- B 链合约与测试：中等
- Reactive 参数调整：中低
- 前端展示改造：中等
- 联调：中等

所以这次升级 **可做，但不要再并行开启其他大方向**。

------

## 5. 是否需要 Sub-agents

默认结论：

- **不需要作为主模式**

原因：

- B 链合约
- Reactive 参数
- 前端展示

三者强耦合，主线程统一推进更稳。

如果一定要用：

- 主 agent：B 链合约 + Reactive + 联调
- sub-agent：前端展示适配

但这不是必须项。

------

## 6. 最终交付判断标准

如果升级完成，应该能做到：

1. 当前策略触发后，B 链不再只显示固定 80% 切仓
2. 页面能显示一笔 mock short 已建立
3. 不同参数输入时，`hedgeSize` 会变化
4. 答辩时可以明确说：
   - “我们不只是自动保护了”
   - “我们已经开始表达自动反向对冲逻辑了”

------

## 7. 一句话执行建议

**新开分支，先做 B 链动态公式，再做 short 语义前端，再做联调；不碰 GMX，不碰预言机，不碰买回。**
